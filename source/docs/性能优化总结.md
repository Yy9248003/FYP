### 2. N+1查询问题修复 ✅

#### 优化前的问题

**PractisesView.getPageInfos**:
- 问题: 循环中访问 `item.project.name` 导致N+1查询
- 问题: 循环中查询 `models.Options.objects.filter(practise__id=item.id).count()` 导致N+1查询
- **优化前**: 1个主查询 + N个project查询 + N个options查询 = 1+2N个查询
- **优化后**: 1个主查询（包含project） + 1个批量options查询 = 2个查询

**TasksView.getStudentTasks**:
- 问题: 循环中访问 `task.project.name` 和 `task.teacher.name` 导致N+1查询
- 问题: 循环中查询 `existingLog` 导致N+1查询
- **优化前**: 1个主查询 + N个project查询 + N个teacher查询 + N个log查询 = 1+3N个查询
- **优化后**: 1个主查询（包含project和teacher） + 1个批量log查询 = 2个查询

**ExamLogsView.getInfo**:
- 问题: 循环中访问 `item.practise.*` 导致N+1查询
- 问题: 循环中查询 `models.Options.objects.filter(id=item.practise.id)` 导致N+1查询
- **优化前**: 1个主查询 + N个practise查询 + N个options查询 = 1+2N个查询
- **优化后**: 1个主查询（包含practise） + 1个批量options查询 = 2个查询

**ExamLogsView.getPageStudentLogs**:
- 问题: 循环中访问 `item.exam.*` 导致N+1查询
- **优化前**: 1个主查询 + N个exam查询 + N个teacher查询 + N个project查询 = 1+3N个查询
- **优化后**: 1个主查询（包含exam、teacher、project） = 1个查询

**ExamLogsView.getPageTeacherLogs**:
- 问题: 循环中访问 `item.exam.*` 和 `item.student.*` 导致N+1查询
- **优化前**: 1个主查询 + N个exam查询 + N个project查询 + N个student查询 = 1+3N个查询
- **优化后**: 1个主查询（包含exam、project、student） = 1个查询

#### 使用的优化技术

1. **select_related**: 用于优化外键查询（一对一、多对一）
   - 示例: `.select_related('project', 'teacher')`
   - 效果: 将多个查询合并为一个JOIN查询

2. **prefetch_related**: 用于优化反向查询（一对多、多对多）
   - 示例: `.prefetch_related('practise__options')`
   - 效果: 使用额外的查询批量获取关联数据

3. **批量查询**: 使用 `__in` 批量获取数据
   - 示例: `models.Options.objects.filter(practise_id__in=practise_ids)`
   - 效果: 将N个查询合并为1个查询

4. **聚合查询**: 使用 `Count` 批量统计
   - 示例: `.values('practise_id').annotate(count=Count('id'))`
   - 效果: 将N个count查询合并为1个查询

## 🔧 技术细节

### select_related 使用示例

```python
# 优化前
tasks = models.Tasks.objects.filter(grade=student.grade)
for task in tasks:
    print(task.project.name)  # N+1查询

# 优化后
tasks = models.Tasks.objects.filter(grade=student.grade).select_related('project', 'teacher')
for task in tasks:
    print(task.project.name)  # 无额外查询
```

### 批量查询示例

```python
# 优化前
for item in items:
    count = models.Options.objects.filter(practise_id=item.id).count()  # N+1查询

# 优化后
practise_ids = [item.id for item in items]
option_counts = models.Options.objects.filter(
    practise_id__in=practise_ids
).values('practise_id').annotate(count=Count('id'))
counts_dict = {item['practise_id']: item['count'] for item in option_counts}
for item in items:
    count = counts_dict.get(item.id, 0)  # 无额外查询
```

## 📝 注意事项

### 数据库迁移

添加索引后需要运行数据库迁移：

```bash
python manage.py makemigrations
python manage.py migrate
```

### 索引维护

- 索引会占用额外的存储空间
- 索引会略微影响INSERT/UPDATE性能
- 但对于查询频繁的字段，索引带来的性能提升远大于维护成本

### 查询优化建议

1. **始终使用select_related/prefetch_related**: 当需要访问外键字段时
2. **避免循环中查询**: 使用批量查询替代
3. **使用only/defer**: 只查询需要的字段
4. **使用values/values_list**: 当只需要部分字段时
